1. 缺乏“依赖关系”与“执行顺序” (Critical)
问题：shot_004 的视频生成依赖于 shot_004 的图片生成结果（input_image: "shot_004.png"）。但在 JSON 中，这只是个字符串，没有显式的依赖图（DAG）。
风险：如果工作流引擎盲目并行执行，可能会在图片还没生成好时就尝试运行 FFmpeg，导致报错。
建议：需要在工作流引擎中内置逻辑：Video 任务必须等待同 Shot 下的 Image 任务成功完成后才能启动。或者在 JSON 中增加 "depends_on": ["tasks.image"] 字段。
2. 缺少“全局资源上下文” (Global Context)
问题：
背景音乐 (BGM)：只有人声（TTS），没有背景音乐的轨道定义。纪录片通常需要铺垫音乐，且音乐需要跨镜头平滑过渡（Fade in/out）。
统一风格约束：虽然每个 prompt 都写了“历史纪录片风格”，但如果想一次性全局切换风格，需要修改几十处。
字体/素材库：标题卡使用的具体字体文件路径、Logo 图片路径未定义。
建议：在 meta 下增加 global_assets（BGM 路径、字体文件、Logo 路径）和 style_preset（全局风格提示词前缀）。
3. 音频混合逻辑缺失 (Audio Mixing Logic)
问题：JSON 定义了 TTS，但没有定义混音策略。
人声音量是多少？
如果有 BGM，BGM 在人声出现时是否自动降低（Ducking）？
音效（SFX，如翻页声、马蹄声）在哪里触发？
现状：目前的结构只能生成一条干声轨道，最后的合成步骤会非常单薄。
4. 错误处理与重试机制 (Error Handling)
问题：AI 生成（尤其是绘图和视频）有失败率（如内容审核拦截、超时、生成质量差）。
风险：如果 shot_005 生成失败了，整个流程是停止？还是跳过？还是自动重试？JSON 中没有定义 retry_policy 或 fallback_prompt（备用提示词）。
5. 文件路径管理的模糊性
问题："input_image": "shot_004.png" 是相对路径还是绝对路径？存放在哪个桶（Bucket）或文件夹？
风险：在分布式环境中（比如图片在 AWS S3，视频生成在另一台机器），简单的文件名无法定位资源。
建议：使用 URI 格式（如 s3://bucket/project/shots/shot_004.png）或由工作流引擎统一管理临时文件映射。
6. 视频生成的“一致性”挑战
问题：shot_007 和 shot_008 都是玄奘。虽然 shot_008 设置了 "reference_image": "shot_007.png"，但这依赖于底层的 seedance 模型是否真的支持并严格执行这个参数。
风险：如果模型忽略参考图，会导致主角长相在前后镜头不一致（Character Consistency 断裂），这是 AI 视频的大忌。JSON 结构无法强制模型遵守一致性，只能依赖模型能力。
7. 转场效果过于单一
问题：目前的 video 任务大多只是简单的 zoom_in（肯尼斯效应）。
不足：专业的纪录片需要复杂的转场（叠化 Dissolve、黑场 Fade to Black、划像 Wipe）。目前 JSON 似乎假设所有镜头直接拼接，缺乏**“镜头间转场（Inter-shot Transition）”**的定义。应该在 shots 数组之间或 meta 中定义转场逻辑。
